Console input parser library

From cppreference.com
< cpp

The console input parser library provides a means of easily and clearly/concisely writing the code which interprets input passed to a program via the console. It does so through the class input_handler, with help from a few other classes/functions defined by the library.

Because of the fact that this library had to be written to work with arbitrary numbers of variables of arbitrary, possibly differing types, it makes heavy use of void pointers and variadic functions. Because of this, compiler-time type-checking is unreliable with this library. Code involving this library is always simple enough that it isn't difficult to find bugs caused by passing bad types, just be aware of the fact that bugs can be caused by passing the wrong type, and that you cannot rely on the compiler automatically converting types for you where void pointers and variadic functions are concerned.

CIP::input_handler

A class which provides a simple, farily comprehensive interface for constructing a command-line argument interpreter

CIP::boolstr

Wraps a boolean with a set of strings mapped to true/false, so that the bool will be set to true or false if an input for it matches a corresponding string.

CIP::intstr

Wraps an int with a set of strings mapped to true/false, so that the int will be set to true or false if an input for it matches a corresponding string.

CIP::function

A templated class which wraps a function pointer with its inputs, so that the function call may be delayed until its inputs are set and/or if/when it is the appropriate time for it to run.
Essentially, it's a means of delaying the excecution of code, so that input_handler can be given code to call at the right time, later.

Exceptions

These exceptions can be thrown by CIP::input_handler::interpretInput(), and are intended to be used for control flow. They provide an intuitive way of representing which code path will be followed given a specific kind of abnormal input.

Using exceptions for control flow is typically considered bad practice, and the fact that these are exceptions causes some inconviniences, so they may be replaced by a different structure in the future.

They all inherit from std::exception, so have a member function "what()" which describes the problem generically. Most of them also have member variables which describe the problem in a way which is easier for a program to parse.

Prebuilt test functions

These are functions which can be wrapped in CIP::functions and provided to input_handler::setTest(). As such, they all take arguments (unsigned int argc, void** argv). On their pages, the arguments which will be listed are those which argv is expected to contain. If such an argument is listed as not being of pointer type, it must be a pointer and is listed as such for the sake of making more intuitive sense

Macros

CIP_ORDERED_START(ih) CIP_ORDERED_END(ih) CIP_FLAGS_START(ih) CIP_FLAGS_END(ih) CIP_TESTS_START(ih) CIP_TESTS_END(ih)
These macros take an an input_handler as argument and call ih.setNum[OrderedVars|Flags|Tests](n) where n = the number of lines between START and END. They call that function on the line where START exists.
The purpose of these macros is to allow the number input to setNumX to change when the number of X changes without you needing to do anything.

They use goto and lables as part of their definition. In an attempt to guarentee that those labels never conflict with one another even in the unexpected case where there are multiple input_handlers, ih (the literal text passed to the macro) is concatenated in the middle of the lables.
CIP_strFs(x) CIP_strs(x)
These macros are used to create the first argment of different overloads of input_handler::setupDynamicInput().
#define CIP_strs(x) new const char*[x]
#define CIP_strFs new CIP::function<std::string>*[x]

Notes

The CIP namespace contains a namespace TYPE which contains integral constants which correspond to the types which input_handler is compatible with.

There are speciallized CIP::function-constructors for each of the prebuilt test and default functions in this library which take inputs specific to each function (allowing for compile-time type-checking, a thing this library can be bad at providing) and call fptr(their_function, those_inputs...).

The naming convention for those prebuilt functions is camelCase. Their constructors keep the same names, but the naming convention changes:

  • they are in UPPER_SNAKE_CASE
  • where there is a standalone capital F in the original name, it is lowercase in the constructor's name. If this forms pseudo-camelCase, no underscores are added (e.g. STRfSUM)
  • there is no constructor for NOT or NOTF
